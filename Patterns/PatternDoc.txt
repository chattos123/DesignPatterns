## singleton

Intent: Ensure a class has only one instance and provide a global point of access to it.
When to use: Use when exactly one object is needed to coordinate actions across the system (e.g., logging, configuration, shared resource manager).
Key points:
- Control instance creation and provide lazy initialization if needed.
- Be careful with multithreading: prefer thread-safe implementations.
- Avoid overuse — singletons are effectively global state.

## AbstractFactory

Intent: Provide an interface for creating families of related or dependent objects without specifying their concrete classes.
When to use: Use when your code needs to work with various families of related products and you want to isolate the creation logic.
Key points:
- Define abstract factory interface and concrete factory implementations.
- Factories return abstract product interfaces.
- Allows swapping entire product families easily.

## Builder

Intent: Separate the construction of a complex object from its representation so that the same construction process can create different representations.
When to use: Use when creating complex objects step by step and the construction process must allow different representations.
Key points:
- Use a Director to orchestrate building steps or let clients call steps directly.
- Builder accumulates state and returns final product with Build/BuildAsync method.

## Factory

Intent: Define an interface for creating an object, but let subclasses decide which class to instantiate. Often implemented as Factory Method or Simple Factory.
When to use: Use when a class can't anticipate the class of objects it must create or wants subclasses to specify created objects.
Key points:
- Keeps creation logic in one place.
- Promotes loose coupling between creator and concrete product classes.

## Prototype

Intent: Specify the kinds of objects to create using a prototypical instance, and create new objects by copying this prototype.
When to use: Use when object creation is expensive or when you want to produce objects by cloning existing instances with modifications.
Key points:
- Implement cloning (shallow or deep) carefully.
- Useful for runtime configuration of objects.

## Adapter

Intent: Convert the interface of a class into another interface clients expect. Adapter lets classes work together that couldn't otherwise because of incompatible interfaces.
When to use: Use when you need to use existing classes but their interfaces do not match the one you need.
Key points:
- Adapter wraps an existing object and translates calls between client and adaptee.
- Can be object adapter (composition) or class adapter (inheritance where supported).

## Bridge
Intent: Decouple an abstraction from its implementation so that the two can vary independently.
When to use: Use when you want to avoid a permanent binding between an abstraction and its implementation, or when both the abstraction and implementation should be extensible by subclassing.
Key points:
- Separate abstraction and implementation into different class hierarchies.
- Use composition to link abstraction with implementation.

## Composite
Intent: Compose objects into tree structures to represent part-whole hierarchies. Composite lets clients treat individual objects and compositions of objects uniformly.
When to use: Use when you want to represent a hierarchy of objects and treat individual objects and compositions uniformly.
Key points:
- Define a common interface for both leaf and composite objects.
- Composite objects can contain other composites or leaf objects.
- Clients interact with objects through the common interface, treating all objects uniformly.

## Decorator
Intent: Attach additional responsibilities to an object dynamically. Decorators provide a flexible alternative to subclassing for extending functionality.
When to use: Use when you want to add behavior to individual objects without affecting other objects of the same class.
Key points:
- Decorators wrap an existing object and provide additional behavior.
- Can be combined in various ways to create complex behavior dynamically.
- Promotes code reuse and adherence to the Open/Closed Principle.

## Observer
Intent: Define a one-to-many dependency between objects so that when one object changes state, all its dependents are notified and updated automatically.
When to use: Use when an object should notify other objects without making assumptions about who these objects are.
Key points:
- Observers register with a subject to receive updates.
- Subject maintains a list of observers and notifies them of state changes.
- Promotes loose coupling between subject and observers.

## Command
Intent: Encapsulate a request as an object, thereby allowing for parameterization of clients with queues, requests, and operations.
When to use: Use when you want to decouple the sender of a request from its receiver, or when you need to support undoable operations.
Key points:
- Command objects encapsulate all information needed to perform an action.
- Supports undo/redo operations by storing state.
- Promotes loose coupling between sender and receiver.

## ChainOfResponsibility	
Intent: Avoid coupling the sender of a request to its receiver by giving more than one object a chance to handle the request. Chain the receiving objects and pass the request along the chain until an object handles it.
When to use: Use when multiple objects can handle a request, and you want to decouple the sender from the receivers.
Key points:
- Each handler decides either to process the request or pass it to the next handler in the chain.
- Promotes flexibility in assigning responsibilities to objects.

## Facade
Intent: Provide a unified interface to a set of interfaces in a subsystem. Facade defines a higher-level interface that makes the subsystem easier to use.
When to use: Use when you want to simplify the usage of a complex subsystem or when you want to decouple clients from the subsystem.
Key points:
- Facade provides a simplified interface to a complex subsystem.
- Clients interact with the facade instead of the subsystem directly.
- Promotes loose coupling between clients and the subsystem.

## Flyweight
Intent: Use sharing to support large numbers of fine-grained objects efficiently.
When to use: Use when you need to create a large number of similar objects that can share common state to save memory.
Key points:
- Flyweight objects share intrinsic state and store extrinsic state externally.
- Use a factory to manage flyweight objects and ensure sharing.
